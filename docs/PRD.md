# 产品需求文档（PRD）

## 1. 产品概述

### 1.1 产品名称

Interview Agent

### 1.2 产品定位

Interview Agent 是一个面向求职训练场景的智能面试问答系统。用户可以上传自己的面试资料、技术笔记、八股文、真实面试复盘与外部参考资料，系统通过 RAG 技术构建用户私有知识库，并在此基础上提供问答、抽题、模拟面试和训练反馈能力。

### 1.3 产品目标

本产品的核心目标不是做一个泛化聊天机器人，而是帮助用户把零散的面试准备材料沉淀为一个可持续增强的训练系统。

核心目标如下：

- 让用户持续积累和管理自己的面试知识资产
- 让系统基于用户资料进行可追溯的 RAG 问答
- 为后续模拟面试 Agent 提供题库、知识库和训练记录基础
- 支持未来扩展到不同岗位方向，而不仅限于 AI Agent 岗位

## 2. 目标用户

### 2.1 核心用户

- 正在准备求职面试的用户
- 希望把八股文、面试笔记和复盘沉淀下来的人
- 需要反复训练某个岗位方向的人

### 2.2 扩展用户

- AI Agent 应用开发岗位求职者
- Java 后端岗位求职者
- 前端岗位求职者
- 测试、算法、产品等其他岗位求职者

## 3. 产品价值

### 3.1 对用户的价值

- 资料不再分散在本地文件、笔记软件和聊天记录中
- 每次面试经历都可以沉淀进知识库，形成持续改进闭环
- 问答基于个人资料，更贴近真实面试准备需求
- 后续可以把系统作为长期自用产品持续迭代

### 3.2 对项目展示的价值

- 可以覆盖 RAG、向量检索、文档解析、异步任务、Agent 编排、前后端分离等关键技术点
- 业务场景具体，适合面试中讲清楚系统设计与取舍
- 能同时体现工程能力和产品思维

## 4. 核心使用场景

### 场景一：上传资料建库

用户上传 `txt`、`md`、`pdf` 等资料，系统解析后构建知识库，供后续检索问答使用。

### 场景二：知识问答

用户围绕某个岗位或知识库提问，系统通过检索相关片段后进行生成回答，并提供引用来源。

### 场景三：面试复盘增强

用户将某次真实面试中遇到的问题、自己的回答和面试官建议整理成文档上传，系统将这些内容纳入知识库，帮助后续训练。

### 场景四：长文自动抽题

用户上传一篇长文，如 RAG、LangGraph、Agent 工程化实践资料，系统自动生成结构化面试题和参考答案。

### 场景五：后续模拟面试

后续版本中，系统会基于用户知识库和题库自动出题，对用户作答进行评分，决定是否追问，并最终给出本轮总结报告。

## 5. 产品范围

## 5.1 一期 MVP 范围

一期必须完成：

- 用户注册与登录
- 多用户数据隔离
- 知识库分类管理
- 文档上传
- 支持 `txt`、`md`、`pdf`
- 异步文档入库
- 文档解析与文本清洗
- 文本切分与向量化
- Chroma 向量索引
- 基于知识库的 RAG 问答
- 回答引用来源展示
- 问答历史记录

一期建议完成：

- 从长文中自动抽题
- 文档元数据提取
- 基础 Prompt 配置抽象
- 按知识库分类检索

## 5.2 二期范围

- 模拟面试工作流
- 出题
- 回答评分
- 追问
- 总结报告
- LangGraph 状态编排
- ReAct Agent 工具调用
- 薄弱点分析
- 错题本

## 6. 功能需求

## 6.1 用户模块

功能要求：

- 支持注册
- 支持登录
- 支持 JWT 鉴权
- 支持获取当前用户信息
- 每个用户只能访问自己的知识库、文档和问答记录

## 6.2 知识库模块

功能要求：

- 创建知识库分类
- 编辑知识库分类
- 删除知识库分类
- 查看知识库分类列表

字段建议：

- 名称
- 描述
- 岗位方向

示例：

- AI Agent
- RAG
- Java 后端
- 面试复盘

## 6.3 文档模块

功能要求：

- 上传文档到指定知识库
- 查看文档列表
- 查看文档处理状态
- 删除文档
- 查看文档基础信息

支持格式：

- txt
- md
- pdf

## 6.4 文档入库模块

功能要求：

- 保存上传文件
- 生成文档记录
- 创建异步任务
- 完成文档解析
- 完成文本清洗
- 完成分块
- 调用 Embedding 模型
- 写入 Chroma
- 更新任务状态与文档状态

## 6.5 问答模块

功能要求：

- 用户选择知识库
- 用户输入问题
- 系统在对应知识库范围内检索相关内容
- 系统基于检索内容生成回答
- 返回引用来源
- 记录问答历史

## 6.6 自动抽题模块

功能要求：

- 用户选择某个文档或知识库
- 系统根据内容生成面试题
- 输出参考答案
- 输出题目标签与难度
- 写入题库表，为二期模拟面试模块预留

## 7. 非功能需求

- 支持多用户
- 文档入库必须异步处理
- 回答尽量可追溯
- 模型供应商可切换
- 为语音模式预留架构扩展点
- 一期避免过度设计和微服务化

## 8. 成功标准

满足以下条件即可认为一期 MVP 达标：

- 新用户可以注册、登录
- 用户可以创建知识库分类
- 用户可以上传文档并看到处理状态
- 文档成功入库后可以被检索
- 用户提问能拿到带引用的答案
- 问答记录可以被保留

## 9. 风险与约束

- 项目周期只有两周，需要强控范围
- PDF 解析质量可能不稳定
- RAG 效果强依赖文档质量和切分策略
- 在线模型与 Embedding 成本需要关注
- 一期不应引入 Django、SpringBoot 或微服务体系

## 10. 产品原则

- 先把主链路做通，再考虑高级能力
- 先保证问答可追溯，再追求 Agent 复杂度
- 先以长期自用价值为导向，再做花哨功能
