# 开发计划文档

## 1. 当前阶段目标

当前项目已经从最初的 RAG MVP，推进到了“问答 + 题目生成 + 面试模拟”的主线阶段。

接下来阶段的目标，不再是继续零散地修前端，而是把“模拟面试训练闭环”做完整：

- 题目可生成
- 面试可发起
- 回答可评估
- 追问可继续
- 总结可沉淀
- 后续训练建议可复用

## 2. 今天已完成的工作

### 2.1 稳定性与通用体验

- 增加了后端统一异常处理和错误日志输出
- 前端全局 API 错误通过 `ElMessage` 弹窗提示
- 优化了 QA 回答逻辑：
  - 有文档上下文时优先基于上下文回答
  - 没有有效参考内容时允许模型自行补充
  - 补充回答统一加前缀 `据我所知，`

### 2.2 前端整理

- 清理并修复多个页面中的中文乱码
- 抽离了部分复用组件，降低后续页面改动成本
- 恢复并整理了文档页、问答页、历史页的结构和文案

### 2.3 题目生成模块

- 题目生成接口已具备真实生成能力
- 题目会持久化到题库
- 当前题目支持基础标签和难度信息
- 已支持的题型标签包括：
  - `concept`
  - `scenario`
  - `followup`
  - `design`

### 2.4 面试模块

- 已完成面试会话和会话轮次持久化
- 已支持多轮追问流程
- 已完成前端面试页面
- 页面风格与现有工作台保持一致

### 2.5 面试总结能力

- 面试结束后会自动生成“面试总结”
- 总结能力做成了可复用的后端能力
- 总结优先走模型生成
- 模型不可用时会回退到本地规则总结
- 总结结果已持久化保存
- 前端页面已展示总结正文与结构化建议

### 2.6 专项面试筛选

- 发起面试时已支持按以下条件筛选题目：
  - 知识库
  - 难度
  - 题型
- 历史记录和详情页已展示题目难度、题型标签

## 3. 当前主线建议

建议接下来继续沿着“面试训练系统”主线推进，而不是回到单纯的页面微调。

推荐顺序如下：

1. 面试配置编排
2. 训练分析沉淀
3. 自动化测试补齐
4. RAG 检索质量增强
5. LangGraph / Agent 化升级

## 4. 下一步建议做什么

### 4.1 第一优先级：面试配置编排

把现在的“按难度 / 题型选题”，继续扩展成更完整的训练配置：

- 指定某个文档来源出题
- 控制是否优先使用最近生成的题目
- 增加“专项训练模式”与“随机训练模式”
- 增加一次训练的题目数量或训练轮数配置

这是下一步最推荐做的内容，因为它能让当前面试模块从“能用”进入“可控、可训练”阶段。

### 4.2 第二优先级：训练分析沉淀

基于已落库的反馈和总结，进一步做：

- 高频薄弱点统计
- 多次面试中的重复问题识别
- 按知识库或题型聚合训练建议
- 面试后的长期复盘页

### 4.3 第三优先级：测试补齐

当前已经完成了编译和构建验证，但还缺少系统化测试。

建议优先补下面这些测试：

- 面试创建接口测试
- 题目筛选逻辑测试
- 追问轮次测试
- 总结生成回退测试
- 题目生成持久化测试

今天已完成的验证包括：

- `python -m compileall backend/app` 通过
- `npm run build` 通过

当前仍存在但未阻塞开发的问题：

## 6. 结束时的项目状态

截至今天结束时，项目已经具备以下演示链路：

1. 上传文档并完成入库
2. 基于知识库进行问答
3. 生成题库题目
4. 发起模拟面试
5. 进行多轮回答和追问
6. 自动生成并查看面试总结
7. 按难度和题型进行专项面试

## 7. 明日恢复开发建议

下次继续开发时，建议按下面顺序恢复：

1. 先阅读 `docs/PROJECT_CONTEXT.md`
2. 再阅读本文件，确认主线任务
3. 直接进入“面试配置编排”
4. 完成后补最小可用测试
5. 再决定是否进入长期训练分析或 Agent 化升级
