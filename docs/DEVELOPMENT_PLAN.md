## 3. 当前最推荐的研发重点

建议接下来把主优先级切换为 RAG 优化，而不是继续扩展训练分析的外围能力。

优先原因：

1. 更贴近面试中的技术深问点
2. 更容易产出有说服力的性能指标和前后对比
3. 能明显增强系统的真实可用性，而不是只增强“流程完整度”

## 4. RAG 优化路线建议

### 4.1 第一优先级：检索质量提升

目标是让系统先“找对内容”，再谈回答质量。

建议顺序：

1. 重新审视 chunk 切分策略
2. 增加 metadata 过滤能力
3. 引入混合检索或重排
4. 补充离线评估集

可执行项：

- 对比不同 chunk 大小、overlap、按段落/标题切分的效果
- 让检索显式利用文档名、章节名、标签等 metadata
- 在向量召回后增加 rerank 层
- 选 20 到 50 个真实问题建立小型评估集，记录命中率和答案质量

### 4.2 第二优先级：答案生成质量与可控性

目标是减少“答非所问”和“引用到了但没答好”的情况。

可执行项：

- 优化 RAG prompt，明确要求先基于证据回答，再补充推断
- 区分“证据充分回答”和“证据不足回答”两套话术
- 调整引用片段拼接方式，避免长上下文噪声
- 让回答结构更固定，例如“结论 + 依据 + 备注”

### 4.3 第三优先级：可观测性与评估

目标是把“感觉变好了”变成“指标证明变好了”。

可执行项：

- 记录每次问答的检索候选、命中文档、分数、最终引用
- 建立最小化 RAG 评估脚本
- 对比优化前后的 top-k 命中、回答完整度、引用准确性
- 为失败案例沉淀 FAQ 和误差分类

### 4.4 第四优先级：面试表达价值高的高级优化

这部分非常适合后续在面试中展开讲。

可执行项：

- 多查询改写：为用户问题生成多个检索视角
- 查询分类路由：区分事实问答、总结问答、对比问答
- 上下文压缩：在召回后先做 evidence selection
- 多阶段回答：先检索，再重排，再生成，最后自检

## 5. 推荐的下一步落地顺序

如果现在正式转入 RAG 优化，建议按下面顺序推进：

1. 建一个最小 RAG 评估集和评估脚本
2. 调整 chunk 策略并做一次前后对比
3. 增加 metadata 过滤和检索日志
4. 引入 rerank 或混合检索
5. 再回头优化 prompt 和回答结构

## 6. 暂缓项

以下内容不是不做，而是暂时不作为最高优先级：

- 弱点文本规范化分类
- 更细粒度的训练画像
- 更复杂的 Agent / LangGraph 编排
- 训练分析页面的复杂可视化

这些方向后续都能继续做，但此刻切入 RAG，会更适合项目整体价值和面试表达。
