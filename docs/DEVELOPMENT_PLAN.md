# 开发计划文档

## 1. 交付目标

在两周内交付一个可演示、可继续开发的 MVP，具备以下能力：

- 用户登录注册
- 知识库分类管理
- 文档上传与异步入库
- 基于知识库的 RAG 问答
- 问答历史

## 2. 开发优先级

优先级从高到低如下：

1. 数据模型稳定
2. 上传到检索主链路打通
3. 问答接口可用
4. 前端具备完整产品感
5. 自动抽题作为亮点增强

## 2.1 当前实际进度校准

目前已经完成：

- 用户认证主链路
- 知识库创建与列表
- 文档上传、入库任务、向量入库
- 问答接口、引用返回、问答会话保存
- 前端 QA 页面真实接口接入
- 前端 History 页面真实会话数据展示
- 文档页、问答页、历史页的乱码与主要模板问题修复

当前仍需收尾：

- 将在线聊天模型接入稳定为默认路径
- 优化问答超时、错误处理和提示行为
- 提炼 QA / History 页中的可复用前端组件
- 提升整体前端一致性和细节交互

## 3. 两周排期

## 第 1-2 天：项目基础设施

后端：

- 初始化 FastAPI
- 配置项目设置
- 配置 PostgreSQL
- 接入 SQLAlchemy 与 Alembic
- 完成用户表和认证模块

前端：

- 初始化 Vue 3 + Vite + TypeScript
- 配置路由
- 配置 Pinia
- 搭建基础布局

输出结果：

- 用户可以注册和登录

## 第 3-5 天：知识库和文档能力

后端：

- 完成知识库 CRUD
- 完成文档表
- 实现文件上传接口
- 保存本地文件
- 创建入库任务记录
- 提供任务状态查询接口

前端：

- 完成知识库管理页
- 完成文档上传页
- 完成文档列表页

输出结果：

- 用户可以创建知识库并上传文档

## 第 6-8 天：RAG 主链路

后端：

- 实现 `txt / md / pdf` 解析器
- 实现文本清洗
- 实现切分逻辑
- 接入 Embedding
- 写入 Chroma
- 实现检索服务
- 实现问答接口

前端：

- 完成问答页面
- 展示回答
- 展示引用片段

输出结果：

- 用户可以基于上传资料提问并得到带引用的回答

## 第 9-10 天：历史记录与稳定性

后端：

- 保存问答会话
- 保存问答消息
- 完善异常处理
- 增加日志

前端：

- 完成问答历史页
- 完善加载态、空态和错误态

输出结果：

- 整体产品链路可用

## 3.1 当前阶段建议执行顺序

结合现有代码状态，建议接下来按以下顺序推进：

1. 联调并稳定 `backend/app/rag/generators` 中的在线模型生成路径
2. 优化问答超时、错误提示与异常恢复体验
3. 将 `frontend/src/views/qa` 与 `frontend/src/views/history` 中重复的会话卡片、消息卡片、引用卡片提炼为复用组件
4. 继续推进 `backend/app/tasks/question_generation_tasks.py` 与相关服务
5. 视效果再补简单 rerank 或 hybrid retrieval

## 第 11-12 天：亮点功能

建议优先做：

- 自动抽题

备选方案：

- 简单 Rerank
- 简单混合检索

推荐理由：

- 自动抽题更贴近产品定位，更适合后续模拟面试扩展

## 第 13-14 天：收尾与展示

- 完善 README
- 补 `.env.example`
- 补接口说明
- 补架构图
- 整理演示脚本
- 准备项目亮点表述

## 4. 开发完成标准

满足以下条件即可认为一期开发完成：

- 用户认证可用
- 知识库管理可用
- 文档上传和任务状态可用
- 向量入库可用
- 问答接口可用
- 引用返回可用
- 历史记录可用

## 5. 范围保护原则

一期不扩展到以下内容：

- 微服务
- 复杂权限体系
- 语音交互
- 完整评分系统
- 大规模平台化管理能力

## 6. 一期完成后的路线

后续建议按以下顺序扩展：

1. 自动抽题常态化
2. LangGraph 面试工作流
3. 回答评分
4. 追问逻辑
5. 面试总结报告
6. 薄弱点分析与长期训练统计
